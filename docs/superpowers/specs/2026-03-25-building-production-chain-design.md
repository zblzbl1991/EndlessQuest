# 建筑生产链与经济循环重设计

## 背景

当前放置类玩法存在结构性问题：
- 资源消耗渠道单一，灵石只有一次性建筑升级消耗，突破零花费
- 建筑与放置玩法脱节，建筑是静态加成器，缺少自动化生产机制
- 灵草/矿材用途窄，不炼东西就无限堆积
- 多个已实现功能存在 bug（生产加成未生效、功法修炼加成失效、秘境不掉装备）

## 设计目标

- 建筑形成互相依赖的生产链，上游产出原料，下游消耗原料产出成品
- 自动化程度：深度自动化，玩家决策在于资源分配和策略配置，非重复点击
- 建筑独立运作，不依赖弟子驻扎分配
- 经济形成闭环：每条路径指向"更强的探索/更高的境界"

## Section 1：生产链核心架构

### 三层建筑体系

| 类型 | 建筑 | 行为 |
|---|---|---|
| **采集层** | 灵石矿、灵田 | 被动产出原料（灵石/矿材/灵气/灵草），受仓库上限约束 |
| **加工层** | 丹炉、锻造坊 | 自动生产队列：持续消耗原料，累积进度，产出成品入库 |
| **辅助层** | 大殿、坊市、藏经阁、聚仙台、传功殿 | 提供加成/解锁功能，不参与生产链 |

### 自动生产队列（核心新机制）

加工层建筑新增自动生产队列：
- 玩家从已解锁配方中选择一个设为"当前生产目标"
- 建筑每秒检查原料库存，足够则消耗原料并累积进度
- 进度达到生产时间则产出成品，尝试进入宗门仓库
- **原料不够时暂停进度**，原料恢复后自动继续
- **仓库（vault）满时也暂停进度**，不消耗原料，有空位后自动继续
- 玩家决策在于"选择生产什么配方"，而非反复点击
- 切换配方时：已累积进度重置为 0，已消耗的原料不退还

### 资源仓库系统

新增资源仓库上限（灵石除外，灵石是通用货币自由流通）：

| 资源 | 初始上限 | 扩展方式 |
|---|---|---|
| 灵气 | 500 | 灵田升级 +300/lv |
| 灵草 | 200 | 灵田升级 +100/lv |
| 矿材 | 200 | 灵石矿升级 +100/lv |

仓库满后采集层建筑停止产出（暂停而非浪费），迫使玩家升级加工层消耗原料或扩展仓库。

## Section 2：资源消耗渠道重设计

### 境界突破消耗丹药 + 灵石

新增丹药消耗，**叠加在**现有渡劫战斗之上（不替换 `breakthroughExtra` 机制）：
- 突破时先检查是否有对应的突破丹 + 足够灵石
- 消耗后进入渡劫战斗（如现有机制）
- 渡劫失败不退还已消耗的丹药和灵石
- 现有 `突破丹`（成功率+20%的 buff 丹药）保留不变，与新突破丹药并行存在

| 境界 | 突破丹需求 | 灵石消耗 |
|---|---|---|
| 练气→筑基 | 筑基丹 ×1 | 500 |
| 筑基→金丹 | 金丹丹 ×1 | 2000 |
| 金丹→元婴 | 元婴丹 ×1 | 8000 |
| 元婴→化神 | 化神丹 ×1 | 30000 |

突破必须消耗丹药（由丹炉生产），形成闭环。

### 探索自动补给消耗

出发时一次性消耗（不是持续消耗），不足则无法选择该补给等级：

| 补给类型 | 消耗 | 效果 |
|---|---|---|
| 基础补给 | 灵石 50 | 解锁探索（不消耗 vault 物品） |
| 精良补给 | 灵石 200 + 回血丹 ×2 | 探索中受伤自动消耗 vault 中的回血丹恢复 HP |
| 豪华补给 | 灵石 500 + 回血丹 ×5 + 突破丹 ×1 | 额外奖励倍率 ×1.5 |

"自动回血"与现有"HP<30% 自动撤退"逻辑共存：精良补给优先消耗回血丹回血，丹药用尽后仍触发撤退。

### 坊市资源交易

新增资源兑换功能（买卖有不对称损耗）：

| 兑换方向 | 汇率 | 损耗 |
|---|---|---|
| 灵石 → 灵草 | 1灵石 = 2灵草 | 无（鼓励用灵石买原料） |
| 灵石 → 矿材 | 1灵石 = 2矿材 | 无 |
| 灵草 → 灵石 | 3灵草 = 1灵石 | 高（不鼓励卖原料） |
| 矿材 → 灵石 | 3矿材 = 1灵石 | 高 |

坊市等级降低卖出损耗：每级降低 5% 损耗率，从基础 66.7%（即 3:1 中 2 份被损耗）降至最低 30%。

公式：`实际获得 = 数量 × (1 - max(0.3, 0.667 - 0.05 × marketLevel))`

### 完整经济闭环

```
灵石矿 ──→ 灵石 + 矿材
                  │               ↕ 坊市交易（有损耗）
灵田   ──→ 灵气 + 灵草          │
    │         │                 │
    │         ├──→ 丹炉 ──→ 丹药 ──→ 突破（核心成长）
    │         │                    ──→ 探索补给
    │         │
    └──→ 弟子修炼（消耗灵气）
              │
    矿材 ──→ 锻造坊 ──→ 装备 ──→ 弟子战力 ──→ 更好探索收益
                                            │
灵石 ──→ 建筑升级 ──→ 更高产出 ──→ 更多资源 ─┘
    ──→ 探索补给 ──→ 探索收益 ──┘
    ──→ 招募弟子 ──→ 更多修炼/探索线 ──┘
```

## Section 3：各建筑具体改造

### 采集层

| 建筑 | 保留 | 新增 |
|---|---|---|
| 灵石矿 | 产出灵石+矿材（被动/秒） | 仓库扩展：矿材 +100/lv |
| 灵田 | 产出灵气+灵草（被动/秒） | 仓库扩展：灵气 +300/lv，灵草 +100/lv |

### 加工层

丹炉配方：

| 丹药 | 解锁条件 | 原料/秒 | 总原料成本 | 生产时间 | 等级门槛 |
|---|---|---|---|---|---|
| 回血丹 | 丹炉 Lv1 | 灵草 0.25 | 灵草 5 | 20s | Lv1 |
| 灵气丹 | 丹炉 Lv1 | 灵草 0.4 | 灵草 10 | 25s | Lv1 |
| 筑基丹 | 丹炉 Lv3 | 灵草 0.5 + 灵石 2 | 灵草 30 + 灵石 120 | 60s | Lv3 |
| 金丹丹 | 丹炉 Lv5 | 灵草 0.5 + 灵石 5 | 灵草 60 + 灵石 600 | 120s | Lv5 |
| 元婴丹 | 丹炉 Lv7 | 灵草 1.0 + 灵石 10 | 灵草 300 + 灵石 3000 | 300s | Lv7 |
| 化神丹 | 丹炉 Lv8 | 灵草 1.5 + 灵石 20 | 灵草 750 + 灵石 10000 | 500s | Lv8 |

配方总原料成本与手动炼制保持一致或略高（自动生产省心但不多占便宜）。
"灵气丹"替代现有"灵气丹"（spirit_potion），统一命名。

锻造坊配方：

| 品质 | 解锁条件 | 原料/秒 | 生产时间 | 等级门槛 |
|---|---|---|---|---|
| 凡品装备 | 锻造坊 Lv3 | 矿材 0.2 + 灵石 0.5 | 30s | Lv3 |

原料/秒：库存够则每秒扣除，累积进度到生产时间产出1个。库存不够或 vault 满时暂停进度。

## Section 4：离线系统与 Bug 修复

### 离线行为

- 自动生产队列离线期间按离线时长结算
- 采用**净速率估算**而非逐秒模拟（避免离线8小时逐 tick 的性能问题）：`产出数量 = min(可用原料 / 原料每秒, vault剩余槽位, 离线时长 / 生产时间)`
- 产出的成品受 vault 上限约束，多余的直接丢弃

### 现有 Bug 修复

| Bug | 修复方式 |
|---|---|
| ResourceEngine 生产加成未生效（tickAll 永远传 {1,1}） | 从功法/弟子计算实际加成传入 |
| cultivationTick 未传入功法参数导致修炼加成失效 | 传入当前功法 |
| 秘境 resolveEvent 不掉装备（itemRewards 永远空） | 添加按层数掉落装备逻辑 |

### 升级成本曲线

当前线性 `baseCost × level` 改为指数 `baseCost × (currentLevel + 1)^1.3`（N 为目标等级，即 currentLevel + 1）。所有建筑的 `baseCost` 保持现有值不变：

| 等级 | 当前（灵石矿） | 新（灵石矿） |
|---|---|---|
| 1→2 | 100 | 100 |
| 3→4 | 300 | 493 |
| 5→6 | 500 | 1,010 |
| 7→8 | 700 | 1,720 |
| 9→10 | 900 | 2,606 |

## Section 5：数据模型与实现边界

### 新增数据结构

```typescript
interface ProductionQueue {
  recipeId: string | null;    // 当前生产配方，null=空闲
  progress: number;           // 当前进度（秒）
}

interface ResourceCaps {
  spiritEnergy: number;       // 灵气上限
  herb: number;               // 灵草上限
  ore: number;                // 矿材上限
}
```

切换配方时进度重置为 0，已消耗原料不退还。

### ResourceCaps 计算方式

加载时从建筑等级动态重建，公式：

```
spiritEnergy_cap = 500 + spiritField_level × 300
herb_cap = 200 + spiritField_level × 100
ore_cap = 200 + spiritMine_level × 100
```

如果加载后资源数量超过上限（不应该发生，但防御性处理），截断到上限值。ResourceCaps 不存储到 save，仅作为运行时计算值在 store 内部使用，UI 显示当前资源时标注上限。

### 自动配方与手动配方的关系

- 自动队列配方（recipes.ts）是**新增的独立数据表**，与现有 AlchemySystem 的 ALCHEMY_RECIPES 并存
- 手动炼制保留：玩家仍可消耗原料立即获得1个成品（应急使用）
- 自动队列配方与手动配方使用相同的配方 ID（如 `hp_potion`），系统根据调用方式区分自动/手动
- 总原料成本：自动生产与手动炼制保持一致或略高（自动省心但不占便宜）

### 数据变更

| 结构 | 变更 |
|---|---|
| `Sect.resources` | 新增运行时 `resourceCaps` 计算值（不持久化） |
| `Building` | 新增 `productionQueue: ProductionQueue` |
| `Sect.vault` | 不变（丹药/装备自动入库走现有逻辑） |
| 配方数据 | 新增 `recipes` 数据表 |

### 实现边界（不在本次范围）

| 排除项 | 理由 |
|---|---|
| 弟子驻扎建筑 | 建筑独立运作 |
| 建筑维护费/俸禄 | 可选后续迭代 |
| 多队列并行生产 | 一栋建筑同时只生产一种配方 |
| 丹药/装备自动分配给弟子 | 产出入库后手动分配 |
| 坊市汇率动态波动 | 固定汇率即可 |

### Save 系统影响

- `ProductionQueue`（配方 ID + 进度）序列化到 save 数据
- `ResourceCaps` 由建筑等级动态计算，加载时重建，不存储
- `recipes` 数据表是静态配置，不走 save
