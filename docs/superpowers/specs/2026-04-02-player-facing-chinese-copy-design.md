# 玩家可见文案全中文化设计稿

**日期**: 2026-04-02  
**状态**: 待审核  
**主题**: 玩家可见文案收口为全中文  
**范围**: `AdventurePage`、`AdventureReportPage`、`BuildingsPage`、探索相关共享组件，以及后续可复用的显示标签层

---

## 一、设计背景

EndlessQuest 当前已经完成多轮玩法、壳层和页面结构调整，整体世界观、视觉语气和信息层级基本稳定在“清墨轻岚”的方向上。但这轮检查里仍然发现一个会直接破坏沉浸感的问题：

1. 页面上还存在零散英文文案，如 `build`、`MAX`、`x`
2. 同一类概念在不同页面里重复手写，后续很容易再次混入英文
3. 某些展示逻辑仍依赖旧英文报告文本，例如 `stable route`

这会带来两个后果：

- 玩家体验断裂，界面不再像完整的仙侠世界，而像开发中的混合版本
- 后续功能继续迭代时，英文文案会不断回流，无法真正收口

因此，本轮目标不是接入完整国际化系统，而是在当前代码结构上补一层足够轻的“玩家可见中文文案层”，把玩家会看到的文本统一收回中文表达。

---

## 二、核心设计结论

本轮采用 **轻量中文文案层**，而不是完整 i18n 改造。

### 2.1 为什么不做完整 i18n

当前项目的核心诉求不是多语言切换，而是：

- 玩家界面必须全中文
- 内部英文代码和枚举可以继续保留
- 这轮改动要尽量少打断现有页面与存档结构

如果直接引入完整 i18n：

- 需要新增更多基础设施
- 需要改写大量现有调用方式
- 对当前“只要求中文落地”的收益不成比例

因此，本轮更合适的做法是：

- 保留英文内部键
- 在显示层集中映射为中文
- 把散落的玩家可见英文逐步收口

### 2.2 本轮的目标状态

完成后应达到以下结果：

1. 玩家可见区域默认不再出现英文单词
2. 重复出现的概念通过共享映射统一输出中文
3. 旧报告、旧存档、旧内部枚举仍可正常工作
4. 新页面后续如果复用同类概念，应优先走共享中文标签，而不是继续手写显示文案

---

## 三、本轮边界

### 3.1 要改的内容

- 页面标题、段落、按钮、标签、提示文案
- 探索结果、战术预设、路线倾向、品质名称等重复显示标签
- 报告页和列表页中仍然直接露给玩家的英文表达
- 关闭按钮、状态标签这类小但明显的界面残留英文

### 3.2 不改的内容

- 类型、枚举、ID、路由参数、存档键名
- 建筑类型、战术值、策略值等内部英文常量
- 测试内部的英文 case 名称
- 非玩家可见的开发注释和实现细节

换句话说：

- `balanced`、`combat`、`recruitmentPavilion` 之类代码值继续保留
- 但它们在界面中展示时，必须经过中文映射

---

## 四、文案层方案

### 4.1 共享显示标签

新增一组轻量共享文案映射，用于承载高频重复概念的中文展示。

优先收口的标签类型包括：

- 探索结果标签
- 战术预设标签
- 路线倾向标签
- 品质标签
- 通用小型界面文案

建议形式：

```ts
export const RESULT_LABELS = {
  completed: '通关',
  retreated: '撤退',
  failed: '失利',
}
```

这类共享映射只负责：

- 把内部英文值映射成玩家看到的中文
- 为多个页面提供同一口径的展示

不负责：

- 承担整站文案管理
- 替代页面内叙述型长句

### 4.2 页面内叙述文案

页面自己的说明句、提示句、描述句仍然保留在页面文件内，但必须改为中文。

例如：

- `关键 build` 改为 `关键构筑`
- `暂无关键 build` 改为 `暂无关键构筑`
- `MAX` 改为 `满级`
- 配方抽屉右上角 `x` 改为更明确的中文关闭动作或语义化图标按钮

原则是：

- 共享概念走映射
- 页面叙述直接写中文

### 4.3 旧报告兼容

当前探索页存在从历史步骤文本中提取路线倾向的逻辑，仍依赖英文关键词：

- `stable route`
- `combat route`
- `profit route`
- `mutation route`

这意味着如果只把新文案改成中文，旧报告和新报告的识别会出现断层。

本轮要求把该逻辑改为：

1. 优先识别中文关键词
2. 同时兼容旧英文关键词
3. 最终统一输出中文路线标签

也就是说：

- 输入可以是旧英文报告
- 输出仍然必须是中文展示

## 五、重点改动区域

### 5.1 AdventurePage

重点问题：

- 战报卡片里直接显示 `关键 build`
- `report.tacticalPreset` 仍直接输出内部英文值
- 历史路线提取逻辑仍依赖英文步骤内容

本页改动要求：

- 所有玩家可见的战术标签改用共享中文映射
- `关键 build` 改为 `关键构筑`
- `暂无关键 build` 改为 `暂无关键构筑`
- 路线识别逻辑兼容中文与旧英文，最终展示中文

### 5.2 AdventureReportPage

重点问题：

- 高光区仍使用 `关键 build`
- `report.config.tacticalPreset` 仍直接暴露内部英文值

本页改动要求：

- 高光区、摘要区统一改为中文标签
- 战术预设改为中文展示
- 保持探索结果、时间线和结算结构不变

### 5.3 BuildingsPage

重点问题：

- 建筑等级后缀仍有 `MAX`
- 自动派驻反馈仍返回英文句子
- 部分细小控件仍带英文交互痕迹

本页改动要求：

- 所有成功/失败消息改为中文
- 建筑等级上限状态改为中文
- 抽屉关闭动作改为中文或中文语义按钮

### 5.4 探索共享组件

涉及组件：

- `RunBuildSummary`
- `TacticPresetPicker`

本轮要求：

- 统一复用共享战术标签
- 保持现有布局和视觉语气不变
- 确保组件单独使用时也不会再露出英文

## 六、实现原则

### 6.1 不改内部结构，只改显示出口

本轮尽量不触碰玩法逻辑与存档结构，而是在“渲染前”完成中文映射。

优先顺序：

1. 先确认文本是否玩家可见
2. 如果是重复概念，抽到共享映射
3. 如果是页面独有句子，直接改为中文

### 6.2 共享映射要小而清楚

本轮不建立巨大文案中心，只建立当前已经高频重复、跨页面共用的显示标签。

这样做的好处是：

- 改动范围可控
- 调用方式简单
- 不会把这轮中文化演变成一轮大重构

### 6.3 历史数据兼容优先于“纯净重写”

凡是已进入旧报告、旧存档、旧记录的英文文本，不要求回写历史数据。

要求的是：

- 新界面读取旧数据时仍能正确识别
- 新界面输出给玩家的结果仍是中文

## 七、测试策略

本轮按 TDD 执行，先补失败测试，再写实现。

### 7.1 测试目标

验证以下结果：

1. 探索页中的关键展示文案已为中文
2. 探索报告页中的高光与战术信息已为中文
3. 建筑页中的状态提示与操作反馈不再出现英文
4. 旧英文路线文本仍能被正确识别，并输出中文路线标签

### 7.2 测试方式

优先补充和更新现有页面测试：

- `AdventurePage.test.tsx`
- `AdventureReportPage.test.tsx`
- 如有必要补充共享组件测试

测试关注点：

- 文本显示是否为中文
- 英文关键词是否从玩家可见区域消失
- 旧英文报告样本是否仍能正常渲染为中文结果

---

## 八、验收标准

本轮完成后，以下条件同时成立才算通过：

1. 当前主流程页面中不再出现玩家可见英文残留
2. 战术、结果、路线、品质等重复概念已有共享中文映射
3. 建筑页与探索页的状态提示、按钮和标签均已收口为中文
4. 旧英文战报文本仍能被兼容识别，不影响展示
5. 实现没有修改内部英文枚举、路由或存档键

---

## 九、后续扩展建议

本轮结束后，不必立刻接入完整 i18n，但应形成一个后续约束：

- 新增玩家可见标签时，先检查是否已有共享中文映射
- 若是重复概念，直接复用
- 若是新概念，再决定是否追加到轻量文案层

这样可以避免项目再次回到“每个页面各写各的，英文又混回来”的状态。
