# 移动端 / 桌面端布局与整体风格审查

**日期**: 2026-04-02  
**审查方式**: 设计文档对照 + 壳层 / 页面样式审查 + 现有截图资产复核  
**范围**: `App` 壳层、`Sidebar` / `TopBar` / `BottomNav`、`Sect` / `Characters` / `Buildings` / `Adventure` / `Vault` / `EventLog` / `AdventureReport`

---

## 一、结论摘要

当前版本的响应式骨架已经成立：

- 移动端底部导航、桌面端左侧侧边栏的切换方向是对的
- 主要页面都具备移动优先的基础结构
- 框架层已经形成统一的清墨雾青外壳，不再像通用后台或 MMO 数值面板

但这一轮审查也确认了 4 个核心问题：

1. **平板区间（640-1023px）还没有被真正设计**  
   多数页面仍沿用 `max-width: 480px` 的手机内容宽度，只是被放到更宽的屏幕里，导致空间浪费，介于“手机放大版”和“桌面版未完成”之间。

2. **桌面端宽屏利用不够一致**  
   有些页面做了并排和多列，有些页面仍然偏窄、偏单列；资源、摘要、二级信息没有稳定地进入“持续可见的桌面上下文”。

3. **整体风格已经转向 `清墨轻岚`，但项目文档仍同时保留“宣纸赭石主导”的旧表述**  
   代码和近期设计稿更接近冷宣纸白、雾青灰、浅墨蓝；项目总文档却仍把“赭石色为强调”写成全局主轴，后续协作容易反复摇摆。

4. **日常页面足够清淡，但“有力量感”的部分还不够集中**  
   日常主视觉 / 卡片 / 面板已经统一，但高光、主操作、关键完成反馈仍偏轻，缺少真正能拉开反差的“落笔点”。

---

## 二、反模式结论

**基本通过，但进入“统一过头”的风险区。**

当前界面没有落入常见的 AI 视觉套路：没有紫蓝霓虹、没有玻璃拟态泛滥、没有居中大数字主视觉，也没有无意义的渐变文字。整体克制、轻、透，方向是健康的。

问题不在“像 AI”，而在于：

- 多个页面复用了非常相似的主视觉 + 摘要 + 分区卡片模板
- 日常卡片的层级差异主要依赖浅色渐变和同一套阴影，重复感开始上升
- 部分语义色仍回退到通用蓝 / 紫后备色，容易破坏世界观统一

换句话说，**它已经不像 AI 套板，但还没有完全长成一套更有记忆点的宗门视觉秩序。**

## 三、布局审查

## 3.1 移动端

### 通过项

- `BottomNav` 固定底部、标签数量可控、触控区达到可用水平，符合碎片时间与拇指操作习惯。
- 各主要页面采用单列结构，信息读取压力基本可控。
- 多个业务弹层已经采用移动端底部抽屉、桌面端居中对话框的双形态思路，方向正确。

### 主要问题

1. **移动端抽屉 / 弹窗策略不一致**  
   `BuildingsPage` 的转移与配方选择更接近“底部抽屉”，但 `CharactersPage`、`AdventurePage` 等仍有居中浮层式交互。  
   结果是同一产品在手机上同时存在两套弹层语言。

2. **资源与页头信息在部分页面偏重复**  
   顶部宗门信息、页面主视觉、局部摘要在移动端偶尔叠加出“同类信息连续出现”的感觉，呼吸感还可以继续压缩和提炼。

3. **高价值操作的视觉力度仍偏轻**  
   主按钮虽已自定义，但在移动端与普通卡片的分离度不算强，尤其在长列表或浅色背景上，主操作没有完全做到“一眼可见”。

## 3.2 平板端（640-1023px）

这是本轮最需要补文档的部分。

### 当前判断

项目文档已经把平板定义为单独阶段，但实现层仍主要围绕 `<640` 与 `>=1024` 两档展开。结果是：

- 壳层仍延续移动端底部导航，这一点合理
- 但页面容器普遍没有放宽到平板应有的阅读宽度
- 列表、摘要、筛选、卡片分组没有稳定提升到 2 列或“主次并排”

### 设计结论

平板不能只是“更宽的手机”。

推荐把 640-1023px 明确视为：

- 继续沿用底部导航
- 页面最大内容宽度放宽到 `680-920px`
- 主视觉 / 摘要 / 筛选 / 列表至少出现稳定的双列组织
- 仍保持触控优先，不引入桌面式密集悬停依赖

## 3.3 桌面端

### 通过项

- 固定侧边栏 + 顶部题首的壳层方向成立，已经具备“宗门卷轴”而非“后台菜单”的雏形。
- `SectPage`、`BuildingsPage`、`CharactersPage` 已经出现多列布局与信息并排，说明桌面增强意识已经建立。
- `AdventureReportPage` 的摘要 + 详情分层比早期版本更清晰。

### 主要问题

1. **不同页面对宽屏的利用强弱不一致**  
   有些页面明显并排，有些页面依然偏单列或较窄，用户切页时会感到密度节奏跳变。

2. **桌面端“持续可见信息”不足够稳定**  
   目前侧边栏只稳定展示宗门名和一项资源，`TopBar` 在桌面端又隐藏资源群，导致宽屏并没有真正换来更强的全局上下文。

3. **“宽屏不浪费”还停留在局部页面，没有成为系统约束**  
   当前实现更像“部分页面已经桌面化”，而不是“整个产品已经有统一的桌面编排规则”。

## 四、整体风格审查

## 4.1 当前风格到底是什么

结合 `theme.css`、壳层组件和 2026-04-01 两份视觉设计稿，当前产品的真实基调已经不是“暖赭旧卷”，而是：

**`清墨轻岚` / `清墨雾青`**

它的特点是：

- 冷宣纸白而不是黄旧纸底
- 雾青灰、浅墨蓝而不是赭石棕作为日常主色
- 整体更轻、更净、更透气

这条路本身是成立的。它解决了早期“偏褐、偏闷、偏旧”的问题，也更符合当前轻提示总览、低操作判断的产品表达。

## 4.2 当前风格的问题不在“错”，而在“分工不够清”

### 已经做对的部分

- 日常壳层足够克制，没有 MMO 式满屏反馈
- 水墨晕染被控制在轻量背景和面板表层，不喧闹
- 字体、导航、卡片语气总体统一

### 还不够好的部分

1. **力量感主要停留在文案，没有完全落到视觉分工上**  
   现在的日常界面很“禅”，但还不够“有力量”。按钮、完成反馈、关键摘要没有形成真正更重的落点。

2. **主强调色太接近日常中性色**  
   当前 `--color-accent` 与大量面板边框、局部高亮过于接近，导致“主操作”“当前页”“重要结论”之间的差距不够明显。

3. **少数语义色跳出世界观**  
   若主题变量缺失时回退到 `#2196f3` / `#9c27b0` 这类通用蓝紫，会把界面瞬间拉向通用后台产品的气质。

4. **重复主视觉模板让页面之间的个性差变弱**  
   当前多个页面共享类似的浅色主视觉结构，统一是有了，但宗门、弟子、建筑、秘境之间的“气口”差异还不够明显。

---

## 五、实现层证据与风险

以下问题应记为“视觉与响应式的一致性债务”：

1. **存在未定义 token 的使用**  
   当前样式中出现了 `--radius-xl`、`--radius-xs`、`--color-text-primary`、`--color-info`、`--color-accent-purple` 等引用，但并未在主题 token 中系统定义。  
   这会让圆角、文字和语义色在不同组件中表现不稳定。

2. **部分页面仍依赖硬编码后备颜色**  
   这会使水墨体系在个别状态下被蓝紫色打断，破坏世界观统一。

3. **移动端 / 桌面端的弹层模式没有完成统一**  
   现有代码已经同时出现底部抽屉与居中浮层两种策略，但缺少明确规范，后续容易继续分叉。

---

## 六、更新后的设计决策

本轮审查后，项目文档统一采用以下决策。

### 6.1 色彩基线

- **日常基线**：采用 `清墨轻岚`  
  即冷宣纸白 + 雾青灰 + 浅墨蓝，保持清淡、透气、克制。
- **力量落点**：赭石 / 印泥色不再承担整个日常主题，而只保留给关键主操作、完成反馈、里程碑与高光摘要。  
  目标是把“力量感”从底色里拿出来，集中到真正重要的时刻。

### 6.2 响应式基线

- `<640px`：单列 + 底部导航 + 底部抽屉优先
- `640-1023px`：仍为触控优先，但内容宽度、信息分组、摘要布局必须显著强于手机
- `>=1024px`：侧边栏常驻，页面至少具备稳定的主次并排能力，不允许只是把手机页面居中放大

### 6.3 壳层基线

- 桌面端必须持续提供“宗门上下文”，而不只是导航
- 移动端必须优先保证拇指触达与单次扫读效率
- 顶部题首、侧边题签、底部标签要继续共用同一套图标与命名世界观

### 6.4 高光与按钮基线

- 日常主按钮仍需强于周围卡片，不能与普通面板同灰同轻
- 关键完成态、突破、通关、稀有掉落、重要离线收益必须明显强于日常壳层
- 不允许再使用脱离主题的通用蓝紫后备色充当“高级”或“稀有”

## 八、审查结论

**当前项目的方向不是推翻重来，而是“继续沿着清墨轻岚的日常基线前进，同时把平板与力量感补完整”。**

如果下一轮工作继续沿用这份结论，产品会更稳定地收束到下面这句话：

**日常像清纸上展开的宗门卷轴，关键时刻才让浓墨与印色真正落下去。**
